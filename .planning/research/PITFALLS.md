# Domain Pitfalls: AICC SaaS 제품화

**Domain:** AI Contact Center (구축형 to SaaS 전환)
**Researched:** 2026-03-31
**Overall Confidence:** MEDIUM (training data + industry pattern knowledge; web verification unavailable)

---

## Critical Pitfalls

실패 시 제품 전체의 방향 수정 또는 재구축이 필요한 수준의 리스크.

---

### Pitfall 1: "구축형 마인드셋의 SaaS 포장" (가장 흔한 실패)

**What goes wrong:**
구축형 솔루션을 UI만 바꿔서 SaaS로 내놓는다. 내부적으로는 고객사별 커스텀 코드, 수동 배포, 전담 엔지니어가 필요한 구조가 그대로 남아 있다. 매출은 SaaS처럼 보이지만 비용 구조는 SI다.

**Why it happens:**
- 기존 구축 납품팀이 제품팀으로 전환되지 못함
- "기존 고객이 요청한 기능"을 그대로 제품에 넣으면서 범용성 상실
- 멀티테넌시가 기술적으로는 지원되지만, 운영 프로세스(배포, 모니터링, 장애 대응)가 테넌트별로 분리되어 있음

**Consequences:**
- 고객 10사만 넘어도 운영 인력이 선형 증가 (SaaS의 레버리지 효과 제로)
- 고객사 A의 핫픽스가 고객사 B에 사이드이펙트 유발
- ARR은 올라가지만 영업이익률은 구축형보다 낮아지는 역설

**Prevention:**
1. **"One Codebase, One Deploy" 원칙 확립** -- 고객사별 분기(fork)가 존재하면 SaaS가 아님
2. **Feature Flag 기반 커스터마이징** -- 고객사별 코드가 아니라 설정값으로 차이를 만듦
3. **고객 요청 = 제품 기능이 아님** -- 3개사 이상 동일 요청이 있을 때만 제품 반영 룰
4. **SaaS Readiness Checklist**: 셀프 프로비저닝, 자동 배포, 공유 인프라, 사용량 계측이 되는지 체크

**Detection (경고 신호):**
- 고객사별 별도 브랜치/환경이 존재
- 신규 고객 온보딩에 엔지니어 2주 이상 투입
- "이 고객사만 쓰는 기능"이 3개 이상

**Phase/Deliverable:** PRD-05 (기존 자산 vs 신규 개발), RM-01 (Phase 1에서 아키텍처 정리 선행)

### Pitfall 2: 보이스봇 "데모 갭" -- 데모에서는 잘 되지만 실전에서 무너짐

**What goes wrong:**
조용한 회의실에서 표준어로 테스트하면 완벽하게 동작하는 보이스봇이, 실제 콜센터 환경에서는 처참한 인식률을 보인다. 고객은 PoC 후 본계약을 거부하거나, 계약 후 조기 해지한다.

**Why it happens:**
- **ASR 환경 차이**: 실제 전화 통화는 8kHz narrowband, 배경 소음, 끊김, 동시 발화가 존재
- **방언/사투리**: 한국어 ASR은 표준어 기준으로 학습되어, 경상/전라/충청 방언에서 에러율 급증
- **도메인 용어**: 금융 전문 용어("변액보험", "신종자본증권"), 이커머스 브랜드명, 축약어를 ASR이 인식 못함
- **대화 흐름**: 고객은 시나리오대로 말하지 않음. 중간에 끊고, 주제를 바꾸고, 불평을 섞음
- **Latency**: 응답이 2초 이상 걸리면 고객이 "여보세요?" 반복 -> 대화 꼬임

**Consequences:**
- 자동 응대율(Containment Rate) 목표 70% vs 실제 30-40%
- 고객사가 2-3개월 만에 "사람이 더 낫다"며 해지
- 엔드유저(소비자) 불만 -> 고객사 브랜드 이미지 하락 -> 계약 파기

**Prevention:**
1. **PoC 환경 = 실전 환경**: 반드시 실제 전화망(PSTN/SIP) 연동, 실제 고객 통화 녹음 데이터로 테스트
2. **도메인 용어 사전 커스터마이징**: 산업별 용어를 ASR 핫워드로 등록하는 셀프서비스 UI 제공
3. **Fallback 설계 필수**: 인식 실패 시 2회 재확인 -> 여전히 실패 시 상담원 즉시 연결 (graceful degradation)
4. **Latency 버짓**: STT + LLM + TTS 전체 파이프라인 1.5초 이내 목표, 초과 시 "잠시만요" 필러 자동 삽입
5. **방언 대응**: 최소한 경상도/전라도 방언 인식률 테스트 -> 부족하면 해당 지역은 상담원 우선 라우팅

**Detection (경고 신호):**
- 데모 성공률 vs 파일럿 성공률 차이 20%p 이상
- "여보세요" 반복 비율 10% 이상 (latency 문제 지표)
- 상담원 전환(escalation) 비율 50% 이상

**Phase/Deliverable:** PRD-03 (보이스봇 MVP), UX-01 (용어 사전 관리 UI), RM-02 (리스크)

### Pitfall 3: 금융/보험 컴플라이언스 지뢰밭

**What goes wrong:**
AICC를 금융/보험사에 배포했는데, 개인정보보호법/전자금융거래법/금융소비자보호법 위반으로 서비스 중단 명령을 받거나, 고객사가 감사 리스크를 이유로 도입을 거부한다.

**Why it happens:**
- **녹음 데이터 처리**: 통화 녹음은 개인정보. 클라우드 저장 시 고객(금융사) 동의 + 암호화 + 보관기간 + 파기 규정 모두 충족해야 함
- **AI 설명 의무**: 금융소비자보호법상 AI가 내린 판단(예: 보험금 산정 안내)은 그 근거를 소비자에게 설명해야 함 -> 블랙박스 LLM으로는 불가
- **데이터 주권**: 금융감독원은 핵심 금융 데이터의 해외 서버 저장을 사실상 금지. AWS Seoul/NCP 등 국내 리전 필수
- **고객 동의**: AI가 상담한다는 사실을 고객에게 고지하고 동의 받아야 함 (2024 개인정보보호법 개정)
- **할루시네이션 법적 리스크**: AI가 잘못된 금리/보험금/약관 정보를 제공하면, 금융사가 법적 책임을 짐

**Consequences:**
- 금융감독원 제재 -> 고객사가 즉시 서비스 중단
- 감사/법무팀 거부로 PoC조차 진입 불가
- 컴플라이언스 대응 비용이 기능 개발 비용을 초과

**Prevention:**
1. **Hybrid RAG의 "출처 추적" 기능을 컴플라이언스 강점으로 포지셔닝** -- "AI가 답변한 모든 내용에 출처(약관 조항, KB 문서 ID)가 태깅됨"
2. **데이터 거주지(Data Residency) 옵션**: 한국 리전 Only 배포 옵션을 v1부터 제공
3. **AI 고지/동의 플로우**: 통화 시작 시 "이 통화는 AI가 보조합니다. 상담원 연결을 원하시면 1번을 눌러주세요" 자동 안내
4. **금융 도메인 가드레일**: 금리, 수익률, 보험금 관련 답변에는 반드시 면책 문구 + 정확한 수치는 공식 문서 링크로 안내
5. **감사 로그(Audit Trail)**: 모든 AI 응답의 입력-출력-출처-타임스탬프를 불변 로그로 보관 (최소 5년)
6. **고객사별 데이터 격리**: 멀티테넌시여도 금융사 데이터는 논리적+물리적 격리 옵션 제공

**Detection (경고 신호):**
- 고객사 법무팀이 검토 단계에서 멈춤
- "감사 대응 자료를 줄 수 있나"는 질문에 즉답 불가
- 통화 녹음이 해외 서버에 저장되고 있음

**Phase/Deliverable:** PRD-01 (포지셔닝에 컴플라이언스 포함), PRD-03/04 (가드레일 기능 명세), RM-01 (Phase 1에서 감사 로그/데이터 격리 선행)

### Pitfall 4: 가격 전략의 "Seat vs Usage" 딜레마

**What goes wrong:**
1. **Per-seat 과금 함정**: 구축형 고객에게 익숙한 라이선스 모델로 SaaS를 팔면, 고객사가 시트 수를 최소화하려고 계정을 공유하거나, "AI가 사람을 대체하면 시트가 줄어야 하는 거 아닌가?"라며 오히려 매출이 감소
2. **Usage-based 과금 함정**: 콜 건수/분당 과금으로 하면, 초기 사용량이 적어 매출이 너무 낮음. 고객도 "비용 예측이 안 된다"며 거부
3. **성과 기반 과금 함정**: "자동 응대 성공 건당 과금"은 매력적이지만, "성공"의 정의가 불명확하고, 초기 정확도가 낮을 때 매출이 0에 가까움

**Why it happens:**
- 구축형 고객은 "한 번 사면 내 거"라는 인식이 강함 -> 월과금 자체에 저항
- AICC의 가치는 "콜 절감"인데, 이걸 직접 매출로 연결하면 고객이 내재화(in-house) 유인이 생김
- 한국 B2B 시장에서 월과금 SaaS 모델 자체가 아직 완전히 정착되지 않음 (특히 금융)

**Consequences:**
- 가격을 너무 낮게 잡으면 LTV가 CAC를 회수 못함
- 가격을 너무 높게 잡으면 구축형 대비 TCO 비교에서 밀림
- 성과 기반 과금에서 "성공"을 둘러싼 고객 분쟁

**Prevention:**
1. **하이브리드 가격 모델**: 플랫폼 기본료(월 고정) + 사용량 기반 변동비 (콜 건수 또는 AI 처리 분 수)
2. **구축형 to SaaS 전환 브릿지**: 기존 구축 고객에게 "구축형 라이선스 비용의 60-70%를 연간 SaaS 비용으로 전환" 마이그레이션 가격 제공
3. **가치 기반 앵커링**: "상담원 1명 인건비 월 300만원" 대비 "AI 보이스봇 월 100만원"으로 프레이밍 -- 절대 "콜 당 X원"만으로 팔지 않음
4. **성과 기반은 프리미엄 티어에만**: 기본 과금은 사용량 기반, 성과 연동은 엔터프라이즈 고객에게 커스텀 계약으로 제공
5. **월 최소 금액(Floor) 설정**: Usage-based라도 월 최소 50만원 등 하한선 설정으로 매출 예측 가능성 확보

**Detection (경고 신호):**
- 파일럿 고객이 "너무 비싸다" vs "너무 싸서 불안하다" 양극단 반응
- 영업팀이 가격표 없이 건건이 협상
- MRR 변동폭이 30% 이상 (usage-based 과금의 위험 신호)

**Phase/Deliverable:** PRC-01/02/03 (가격 전략 전체), PRD-07 (KPI에 ARPU, Churn Rate 포함)

## Moderate Pitfalls

제품 성장을 더디게 하거나 기술 부채를 쌓는 리스크.

### Pitfall 5: 셀프 온보딩의 환상

**What goes wrong:**
"SaaS니까 셀프 온보딩"을 목표로 만들었는데, AICC 제품은 설정할 게 너무 많아서 고객이 혼자 세팅을 완료하지 못한다. 가입 -> 첫 통화 성공까지 도달률이 10% 미만.

**Why it happens:**
- AICC 온보딩에는 PBX/SIP 연동, 지식베이스 구축, 시나리오 설계, 테스트 통화가 필요
- 금융/보험 고객사의 IT팀은 보안 심의 -> 네트워크 개방 -> 연동 테스트에 최소 2-4주 소요
- "셀프"라 해도 결국 CS팀이 수동 가이드 -> 구축형과 비용 차이 없음

**Prevention:**
1. **"Guided Self-Service"**: 완전 셀프가 아니라, 온보딩 위자드 + CSM(Customer Success Manager) 비디오콜 1회 조합
2. **타임 투 밸류(TTV) 최소화**: 가입 후 15분 이내에 데모 시나리오 + 테스트 통화 1건을 체험할 수 있는 샌드박스 환경 제공
3. **산업별 템플릿**: "금융 FAQ 봇", "이커머스 주문조회 봇" 등 사전 구성된 템플릿으로 시작 -> 커스터마이징은 이후

**Detection (경고 신호):**
- 가입 후 7일 내 첫 통화 성공 비율 30% 미만
- 온보딩 도중 CS 티켓 3건 이상 발생

**Phase/Deliverable:** UX-03 (셀프 온보딩 화면), PRD-06 (온보딩 우선순위)

### Pitfall 6: Agent Assist의 "알림 피로" (Alert Fatigue)

**What goes wrong:**
Agent Assist가 너무 많은 추천/알림을 실시간으로 띄워서, 상담원이 오히려 방해받는다고 느낀다. 추천 정확도가 70% 미만이면 상담원이 기능을 끄고 무시한다.

**Why it happens:**
- "기능이 많을수록 좋다"는 전제로 실시간 요약 + 감정 분석 + 추천 답변 + 고객 정보 + 규정 알림을 동시에 표시
- 상담원은 통화 중 인지 부하가 이미 높음 -- 추가 정보 처리 여력이 제한적
- 추천 답변이 맥락에 안 맞으면 신뢰도 급락 -> 전체 기능 불신

**Prevention:**
1. **"Less is More" 원칙**: 통화 중에는 핵심 정보 1-2개만 표시, 나머지는 접힌 상태
2. **상담원 피드백 루프**: "이 추천이 도움이 됐나요?" 원클릭 피드백 -> 추천 품질 지속 개선
3. **점진적 노출**: Phase 1에서는 실시간 요약 + 고객 정보만, Phase 2에서 추천 답변, Phase 3에서 감정 분석 순서로 추가
4. **추천 정확도 80% 이상 확보 후 출시**: 부정확한 추천은 없는 것보다 나쁨

**Detection (경고 신호):**
- 상담원의 Agent Assist 기능 비활성화 비율 30% 이상
- 추천 답변 클릭률 10% 미만
- "방해된다"는 피드백 반복

**Phase/Deliverable:** PRD-04 (Agent Assist MVP), UX-02 (실시간 보조 화면 -- 정보 밀도 설계)

### Pitfall 7: LLM 할루시네이션이 고객사에 법적 책임을 전가

**What goes wrong:**
Hybrid RAG를 쓰더라도, LLM이 지식베이스에 없는 내용을 자신있게 답변하는 경우가 발생한다. 금융 상품의 수익률, 보험 보장 범위 등을 잘못 안내하면 고객사가 법적 분쟁에 휘말린다.

**Why it happens:**
- RAG retrieval 단계에서 관련 문서를 못 찾으면, LLM이 학습 데이터 기반으로 "그럴듯한" 답변 생성
- Confidence threshold가 없거나 너무 낮게 설정됨
- 프롬프트 가드레일이 불충분함

**Prevention:**
1. **답변 불가 판단 로직**: RAG 검색 결과의 relevance score가 임계값 미만이면, "정확한 확인을 위해 전문 상담원에게 연결해드리겠습니다" 자동 응답
2. **출처 필수 정책**: 모든 AI 답변에 출처 문서 ID 태깅 -> 출처 없는 답변은 차단
3. **금융 민감 토픽 블랙리스트**: 금리, 수익률, 보험금 산정 등은 AI 직접 답변 금지 -> 공식 문서 링크 + 상담원 연결
4. **정기 품질 테스트**: 월 1회 adversarial test (의도적으로 어려운 질문) 자동화

**Detection (경고 신호):**
- RAG 미검색(no retrieval) 답변 비율 5% 이상
- 고객 클레임 중 "잘못된 안내" 관련 건 발생
- 출처 태그 없는 답변 비율 증가

**Phase/Deliverable:** PRD-03 (보이스봇 가드레일), PRD-01 (Hybrid RAG 포지셔닝)

---

### Pitfall 8: 멀티테넌시의 "노이즈 네이버" 문제

**What goes wrong:**
한 고객사의 대량 콜 유입이 다른 고객사의 응답 속도에 영향을 준다. 또는 한 고객사의 지식베이스 업데이트가 공유 인프라의 성능을 저하시킨다.

**Why it happens:**
- 구축형에서는 고객사별 전용 인프라였기 때문에 리소스 경합이 없었음
- SaaS로 전환하면서 비용 절감을 위해 공유 인프라를 쓰지만, 격리 수준이 불충분
- LLM API 호출은 특히 rate limit에 민감 -- 한 테넌트가 burst하면 다른 테넌트가 throttle됨

**Prevention:**
1. **테넌트별 Rate Limiting**: 각 고객사에 API 호출 쿼터 할당
2. **큐 기반 격리**: 고객사별 메시지 큐 분리 -> 한 고객사의 burst가 다른 고객사에 영향 안 줌
3. **엔터프라이즈 전용 인프라 옵션**: 금융/대형 고객사에는 dedicated cluster 옵션 제공 (프리미엄 가격)
4. **LLM 호출 캐싱**: 동일 질문에 대한 응답 캐싱으로 API 부하 감소

**Detection (경고 신호):**
- 특정 시간대에 특정 고객사만 latency 증가
- LLM API rate limit 에러 빈발

**Phase/Deliverable:** PRD-05 (인프라 아키텍처), RM-01 (Phase 1 인프라 설계)

## Minor Pitfalls

인지하고 관리하면 크게 문제되지 않지만, 무시하면 누적되는 리스크.

---

### Pitfall 9: 한국어 TTS의 "로봇 느낌"

**What goes wrong:**
기술적으로는 동작하지만, TTS 음성이 부자연스러워서 고객이 "기계랑 말하기 싫다"며 상담원 전환을 요청한다. 특히 금융/보험은 신뢰감이 중요한 도메인이라 영향이 큼.

**Prevention:**
1. 고품질 한국어 Neural TTS 사용 (Google Cloud TTS, NAVER CLOVA Voice, KT AI 등 비교 필요)
2. 음성 톤/속도를 산업별로 커스터마이징 (금융: 신뢰감, 이커머스: 친근함)
3. 긴 문장을 자연스럽게 끊어 읽는 SSML 마크업 적용

**Phase/Deliverable:** PRD-03 (보이스봇 음성 품질 요구사항)

---

### Pitfall 10: 경쟁사 대비 "기능 체크리스트 경쟁" 함정

**What goes wrong:**
Genesys, 카카오엔터프라이즈, 솔트룩스 등과 기능 비교표에서 이기려고 기능을 무한정 추가한다. 결과적으로 어떤 기능도 깊이가 없는 "jack of all trades" 제품이 된다.

**Prevention:**
1. **뾰족한 강점 1개에 집중**: "컴플라이언스 세이프한 금융 AICC" 하나로 포지셔닝
2. **기능 수 제한**: MVP에서 핵심 기능 5개 이내, 나머지는 "Coming Soon"
3. **경쟁 비교 시 카테고리 재정의**: 범용 CCaaS가 아니라 "규제 산업 특화 AICC"로 카테고리 자체를 새로 만듦

**Phase/Deliverable:** MKT-01/03 (경쟁사 분석, 차별화), PRD-06 (기능 우선순위)

---

### Pitfall 11: 기존 구축 고객의 SaaS 전환 저항

**What goes wrong:**
기존 구축 납품 고객에게 "앞으로는 SaaS만 합니다"라고 하면, 고객이 이탈하거나 기존 계약 갱신을 거부한다. 특히 금융사는 "우리 데이터가 다른 회사와 같은 서버에?"라며 거부.

**Prevention:**
1. **병행 운영 기간**: 최소 12-18개월은 구축형과 SaaS를 병행 제공
2. **구축형 고객 전환 인센티브**: SaaS 전환 시 첫 6개월 할인, 마이그레이션 지원 무료
3. **Private SaaS 옵션**: 금융사를 위한 전용 클러스터 SaaS (인프라는 공유하지 않지만 소프트웨어는 SaaS 업데이트)

**Phase/Deliverable:** PRC-01 (구축형 가격), PRC-02 (SaaS 가격), RM-01 (전환 전략)

## Phase-Specific Warnings

| Phase/Topic | Likely Pitfall | Mitigation | Severity |
|---|---|---|---|
| **PRD (제품 정의)** | 구축형 경험을 그대로 PRD에 반영 -> 커스텀 SI 제품이 됨 | "3개사 이상 공통 니즈"만 제품 기능으로 반영하는 원칙 수립 | Critical |
| **보이스봇 MVP** | 데모 환경 vs 실전 환경 갭으로 PoC 실패 | 실전 환경 테스트(실제 전화망, 실제 소음)를 PoC 전에 내부적으로 완료 | Critical |
| **Agent Assist MVP** | 기능 과다로 상담원이 오히려 기피 | Phase 1은 실시간 요약 + 고객 정보만, 추천 답변은 정확도 검증 후 Phase 2 | Moderate |
| **온보딩 설계** | 셀프 온보딩 목표 but 실제로는 CS팀 풀가동 | Guided self-service + 산업별 템플릿으로 TTV 최소화 | Moderate |
| **가격 전략** | Per-seat으로 갈지 usage로 갈지 결정 못하고 혼선 | 기본료(고정) + 사용량(변동) 하이브리드 모델로 확정, 성과 기반은 엔터프라이즈만 | Critical |
| **컴플라이언스** | 금융감독원 규제 미대응으로 파일럿 진입 불가 | 감사 로그, 데이터 격리, AI 고지/동의 플로우를 Phase 1 필수 기능으로 포함 | Critical |
| **인프라 (멀티테넌시)** | Noisy neighbor로 SLA 미달 | 테넌트별 rate limit + 큐 격리, 대형 고객 전용 클러스터 옵션 | Moderate |
| **시장 포지셔닝** | 기능 체크리스트 경쟁에 빠짐 | "규제 산업 특화 AICC" 카테고리 재정의, 뾰족한 1개 강점 집중 | Moderate |
| **기존 고객 전환** | 구축 고객 이탈 | 12-18개월 병행 운영 + Private SaaS 옵션 | Moderate |

## Summary: Top 5 "Kill the Product" Risks

1. **구축형 마인드셋 SaaS 포장** -- 고객 수가 늘어도 비용이 선형 증가하면 SaaS의 의미 없음
2. **보이스봇 데모 갭** -- PoC 실패 = 시장 신뢰 상실, 회복 불가
3. **금융 컴플라이언스 미대응** -- 타겟 시장 자체에 진입 불가
4. **가격 전략 실패** -- 수익 모델이 안 되면 제품이 아무리 좋아도 사업 불가
5. **LLM 할루시네이션** -- 금융/보험에서 한 건의 오답 = 법적 분쟁 + 고객사 이탈
