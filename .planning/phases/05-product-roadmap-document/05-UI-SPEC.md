# Phase 5 -- UI Design Contract (Document Production)

> Visual and formatting contract for Markdown document deliverables. Generated by gsd-ui-researcher.
>
> **Phase 5 is a document authoring phase.** Deliverable is `docs/product-roadmap.md` (RM-01, RM-02). There are no React components, screens, or interactive UI elements. This contract defines the **document formatting patterns and ASCII visualization standards** that ensure visual consistency with Phase 1~4 outputs. Phase 5 is the final document in the planning set and must demonstrate cross-document consistency with all prior deliverables.

## Spacing Scale (Document Hierarchy)

Markdown documents use whitespace and heading levels instead of pixel spacing. The following contract ensures consistent visual rhythm, continuing Phase 3 UI-SPEC conventions.

| Token | Markdown Equivalent | Usage |
|-------|-------------------|-------|
| Section break | 2 blank lines before `## H2` | Major topic transitions (e.g., "Phase별 기능 배포 계획" to "기술 의존성 그래프") |
| Subsection break | 1 blank line before `### H3` | Subtopic transitions (e.g., "MVP Phase 1" to "확장 Phase 2") |
| Paragraph gap | 1 blank line between paragraphs | Standard prose separation |
| List item gap | No blank lines between items | Tight lists for scan-readability |
| Table buffer | 1 blank line before and after table | Visual separation of structured data |
| Blockquote buffer | 1 blank line before and after `>` block | Citation/data point isolation |
| ASCII diagram buffer | 2 blank lines before and after ` ``` ` code block | ASCII 간트 차트, 의존성 그래프, 리스크 그리드의 시각적 분리 |

Heading levels (exactly 3, matching Phase 1~4):

| Level | Usage | Example |
|-------|-------|---------|
| `# H1` | Document title only (1 per file) | `# 뤼튼 AICC SaaS 제품 로드맵` |
| `## H2` | Major sections (numbered: 1, 2, 3...) | `## 2. Phase별 기능 배포 계획` |
| `### H3` | Subsections (numbered: 2.1, 2.2...) | `### 2.1 MVP (Phase 1): Day 1 신뢰성 확보` |
| `#### H4` | Rare: sub-subsection or table title within ASCII diagram context | `#### 마일스톤별 KPI 목표 매핑` |

Exceptions: none

## Typography (Document Formatting)

Markdown rendering is font-agnostic. This contract defines emphasis and formatting patterns, continuing Phase 3 UI-SPEC conventions.

| Role | Formatting | Usage |
|------|-----------|-------|
| Body text | Plain paragraph | Korean prose, 1-3 sentences per paragraph for scannability |
| Key metric | `**bold**` wrapping | Numbers, percentages, periods: `**18개 기능**`, `**6-9개월**`, `**₩2,000/건**` |
| Technical term | Plain (first use) or backtick for code-like terms | `Hybrid RAG`, `STT/TTS`, `PRD-06` |
| Emphasis | `**bold**` | Strategic conclusions, product names at first mention |
| Cross-reference | `(PRD-XX 참조)` or `(MKT-DP-XX 참조)` inline | 선행 문서 데이터 포인트 참조 |
| Table header | Pipe table with left-alignment | Left-align text, left-align numbers (Korean convention) |
| ASCII diagram label | Code block (` ``` `) | 간트 차트, 의존성 그래프, 리스크 매트릭스 그리드 |

**Language rules (from CLAUDE.md):**
- Body text: Korean
- Technical terms, variable names, product names: English where standard (e.g., "Hybrid RAG", "Agent Assist", "MVP")
- Currency: `₩` prefix for Korean Won
- Large Korean Won amounts: `만원` units (e.g., `₩200만`)
- Phase/timeline references: `M1`, `M2`... `M18` 형태 (Month 1~18)

## Color (Document Formatting)

Markdown documents have no color control. Visual hierarchy is achieved through formatting weight. Continuing Phase 3 UI-SPEC conventions.

| Role | Formatting Equivalent | Usage |
|------|----------------------|-------|
| Dominant (60%) | Plain body text | Narrative prose, explanations, rationale |
| Secondary (30%) | Tables + ASCII diagrams + blockquote citations | Structured data, visual charts, evidence citations |
| Accent (10%) | **Bold key numbers** + bullet highlights | Critical metrics, strategic conclusions |
| Destructive | Not applicable | No destructive actions in document phase |

Accent (bold) reserved for:
- Team size numbers (e.g., `**10-12명**`)
- Timeline durations (e.g., `**6-9개월**`)
- Feature counts (e.g., `**MVP 18개 기능**`)
- KPI target numbers (e.g., `**해결률 35%**`, `**할루시네이션 <3%**`)
- Price points (e.g., `**₩2,000/건**`)
- Risk severity indicators (e.g., `**높음**`)
- Strategic conclusion phrases (e.g., `**성과 과금 + 컴플라이언스 엔진이 Day 1부터 포함**`)

Bold must NOT be used for: generic emphasis, section headers (headers self-emphasize), table cell contents (table structure provides emphasis), ASCII diagram internal labels (code block provides emphasis).

## ASCII Visualization Contract

Phase 5의 핵심 산출물은 3종의 ASCII 시각화다. Phase 4에서 확립된 고밀도 ASCII 표현 패턴을 계승한다.

### Visualization 1: 기술 의존성 그래프

**요구사항 (CONTEXT.md):** 노드-엣지 ASCII 플로우차트, 기능 모듈 단위, Build/Enhance 구분 표시

**노드 표현 규칙:**

| 요소 | ASCII 패턴 | 예시 |
|------|-----------|------|
| 기능 모듈 노드 | `┌─────────────┐` / `│ 모듈명       │` / `└──────┬──────┘` | 박스 + 수직/수평 연결선 |
| Build/Enhance 태그 | `[Build]` 또는 `[Enhance]` 노드 우측 또는 내부 | `│ 할루시네이션 가드 │ [Build][L]` |
| T-shirt 사이징 | `[S]`, `[M]`, `[L]`, `[XL]` | 태그 바로 옆에 배치 |
| 의존성 화살표 | `│`, `▼`, `─`, `▶`, `┌`, `┐`, `└`, `┘` | Unicode box-drawing characters |
| 분기 | `┌──┼──┐` 형태 | 하나의 모듈에서 여러 모듈로 분기 |

**필수 포함 노드 (PRD-05/06 기반):**
- STT/TTS 엔진 [Enhance][S]
- 대화 관리 [Enhance][S]
- Hybrid RAG 응답 [Enhance][M]
- 할루시네이션 가드 [Build][L]
- 컴플라이언스 엔진 [Build][L]
- 성과 과금 엔진 [Build][L]
- Agent Assist 실시간 전사/요약/추천 [Enhance][S-M]
- 셀프 온보딩 [Enhance][M]
- 관리 콘솔 [Enhance][M]

**크리티컬 패스 표시:** 할루시네이션 가드 > 컴플라이언스 엔진 > 성과 과금 엔진 직렬 경로를 시각적으로 강조 (두꺼운 선 또는 `═══` 이중선)

### Visualization 2: ASCII 간트 차트

**요구사항 (CONTEXT.md):** 월별 타임라인, 기능 모듈 바 형태, KPI 목표치 연결

**간트 차트 표현 규칙:**

| 요소 | ASCII 패턴 | 예시 |
|------|-----------|------|
| 타임라인 헤더 | 고정폭 월 단위 `M1  M2  M3...` | 최소 M1~M18 범위 |
| 기능 바 | `████` (진행 중) | 월별 칸에 블록 문자로 기간 표시 |
| 마일스톤 마커 | `↑ 마일스톤명` 하단 행 | 간트 차트 아래 마일스톤 행 |
| 구분선 | `─────────────` | 헤더/본문/마일스톤 구분 |
| Phase 구분 | `│` 수직선 | MVP/확장/고도화 Phase 경계 |

**필수 마일스톤 (CONTEXT.md + PRD-07):**

| 마일스톤 | 시점 (추천) | KPI 연결 |
|---------|-----------|----------|
| 알파 테스트 | M3-M4 | 내부 기능 검증 |
| 베타 테스트 | M6-M7 | 파일럿 고객 투입 |
| MVP 출시 | M8-M9 | 해결률 35%, 할루시네이션 <3% (PRD-07 90일 목표) |
| 안정화 | M12 | 해결률 50%, AHT 단축 25% (PRD-07 180일 목표) |
| 확장 완료 | M15-M18 | 해결률 65%, 할루시네이션 <0.5% (PRD-07 1년 목표) |

### Visualization 3: 리스크 매트릭스 3x3 그리드

**요구사항 (CONTEXT.md):** 기술/시장/조직 3차원 분류, 3x3 확률-영향도 그리드, 완화 전략 + 담당 + 트리거

**그리드 표현 규칙:**

| 요소 | ASCII 패턴 |
|------|-----------|
| 그리드 셀 | `│` 수직선 + `───` 수평선 |
| 축 라벨 | 좌측: 확률(높/중/낮), 상단: 영향도(낮/중/높) |
| 리스크 ID | `[T-N]` 기술, `[M-N]` 시장, `[O-N]` 조직 |
| 고위험 셀 (우상단) | 확률 높 x 영향 높 -- 가장 먼저 표시 |

**리스크 상세 표 필수 칼럼:**

| ID | 리스크 | 차원 | 확률 | 영향 | 완화 전략 | 담당 | 트리거 |

**필수 리스크 항목 (CONTEXT.md + market-analysis.md):**
- [T] LLM 할루시네이션 -- Phase 1 시장 분석에서 식별된 핵심 기술 리스크
- [T] 음성 인식 정확도 -- 보이스봇 코어 품질 리스크
- [M] KT 69% 독점 시장 관성 -- MKT-DP-04 직접 참조
- [M] 컴플라이언스 규제 환경 변화 -- 금융위 가이드라인 등
- [O] AI/ML 인력 확보 -- 가상 팀 구성의 실행 리스크

### Cross-Document Consistency Rules (Phase 5 Critical)

이 Phase의 본질은 "새로운 정보 생산"이 아니라 "기존 정보의 전략적 배열과 시각화"다. 다음 일관성 규칙을 반드시 준수한다.

1. **PRD-06 매핑 완전성:** PRD-06 우선순위 매트릭스의 MVP 18개 기능 + Phase 2 2개 기능이 로드맵 Phase별 배포 계획에 전수 매핑되어야 한다. 기능 수, 기능 이름, Build/Enhance 태그, T-shirt 사이징이 정확히 일치해야 한다.
2. **PRD-05 자산 태그:** 의존성 그래프의 각 노드에 PRD-05 Build/Enhance 분류가 표시되어야 한다.
3. **PRD-07 KPI 일치:** 마일스톤의 KPI 목표 수치가 PRD-07의 90일/180일/1년 목표와 정확히 일치해야 한다.
4. **가격 전략 포함:** 성과 과금 엔진과 컴플라이언스 엔진이 MVP (Phase 1)에 반드시 포함되어야 한다 (CONTEXT.md 잠금 결정).
5. **시장 데이터 참조:** 리스크 매트릭스의 시장 리스크 항목이 Phase 1 market-analysis.md의 데이터 포인트(MKT-DP-XX)를 직접 참조해야 한다.
6. **인용 형식:** 기존 참조는 `(PRD-XX 참조)`, `(MKT-DP-XX 참조)` 인라인 스타일. Phase 3에서 확립한 `[KPI-BP-XX]`, `[PRC-BP-XX]` 태그도 필요 시 참조.
