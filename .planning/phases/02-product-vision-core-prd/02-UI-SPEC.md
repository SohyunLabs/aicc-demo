# Phase 2 — UI Design Contract (Document Edition)

> Visual and formatting contract for the PRD document (`docs/prd-core.md`). Phase 2 produces a Markdown document, not UI code. This contract defines the document's visual structure, formatting patterns, and copywriting standards to ensure consistency with Phase 1 outputs and readability for the target audience (leadership/executives).

## Spacing Scale (Document Equivalent)

Markdown documents use vertical whitespace (blank lines) and heading hierarchy for spacing. These rules ensure consistent document rhythm.

| Token | Markdown Equivalent | Usage |
|-------|---------------------|-------|
| xs | No blank line | Inline elements, list item continuation |
| sm | 1 blank line | Between paragraphs within a subsection |
| md | 1 blank line + `---` (horizontal rule) | Between major sections (Section 1, 2, 3, 4) -- NOT used |
| lg | 2 blank lines | Between top-level sections (## headings) |
| xl | 2 blank lines + newline | Between major document parts (Executive Summary, Sections, Appendix) |

**Document-specific rules:**
- Between `##` sections: 2 blank lines before the heading (consistent with Phase 1 `docs/market-analysis.md`)
- Between `###` subsections: 2 blank lines before the heading
- Between `####` sub-subsections: 1 blank line before the heading
- No horizontal rules (`---`) between sections (Phase 1 did not use them within the body)
- Blockquote data points (`> **[MKT-DP-XX]**`) separated by 1 blank line from surrounding text

Exceptions: none

## Typography (Document Heading Hierarchy)

Markdown documents use heading levels instead of font sizes. This contract maps heading hierarchy to document structure.

| Role | Markdown | Usage | Line Length Limit |
|------|----------|-------|-------------------|
| Document Title | `#` (H1) | Document title only -- exactly 1 per document | 80 characters |
| Section | `##` (H2) | 5 sections: Executive Summary, Section 1-4, Appendix | 60 characters |
| Subsection | `###` (H3) | Subsection within each section (e.g., 1.1, 2.1, 3.1) | 50 characters |
| Feature/Card | `####` (H4) | Individual feature entries, persona cards, scenarios | 50 characters |

**Text formatting conventions:**
- Body text: Plain Markdown, no indentation, Korean language
- Bold (`**text**`): Reserved for emphasis on key terms, data points, product names
- Italic (`*text*`): Not used (consistency with Phase 1)
- Inline code (`` `text` ``): Reserved for technical terms when necessary (e.g., `Hybrid RAG`, `STT/TTS`)
- Blockquote (`>`): Reserved for data point citations (`> **[MKT-DP-XX]**`)
- Line length: Soft limit 120 characters per line for readability

**Weight convention (bold usage):**
- Numbers/statistics: Always bold (e.g., **3.5조원**, **69%**, **23.7%**)
- Product/brand names on first mention: Bold (e.g., **Perplexity**, **KT**)
- Key terms on first definition: Bold
- Do not bold entire sentences or paragraphs

## Color (Document Visual Hierarchy)

Markdown documents rely on structural elements rather than color. This contract defines the visual weight hierarchy.

| Role | Markdown Element | Visual Weight | Usage |
|------|------------------|---------------|-------|
| Dominant (60%) | Plain body text | Normal | Explanatory prose, analysis |
| Secondary (30%) | Tables (`\| \|`) and blockquotes (`>`) | Medium | Data comparisons, persona cards, feature matrices, data point citations |
| Accent (10%) | Bold text (`**`) + emoji-free callout patterns | High | Key statistics, critical terms, Day 1 must-have labels |

**Accent reserved for:**
- Statistical data points (market size, growth rates, percentages)
- Day 1 mandatory feature labels
- Core value connection phrases ("틀리면 안 되는 상담")
- Product differentiator terms (출처 추적, 할루시네이션 가드, 컴플라이언스)

**Destructive equivalent:** Not applicable (no destructive actions in a document)

**Visual hierarchy enforcement:**
- No emoji usage anywhere in the document (executive audience, professional tone)
- No HTML tags within Markdown (pure Markdown only)
- No color-coded badges or shields

## Document Structure Contract

### Page Budget (10페이지 이내)

| Section | Allocated Pages | Content Type |
|---------|----------------|--------------|
| Executive Summary | 0.5p | Perplexity 비유 hook + 핵심 메시지 3줄 |
| Section 1: 비전/포지셔닝 (PRD-01) | 1.5p | 시장 기회 + 구축형 vs SaaS + Perplexity 비유 |
| Section 2: 페르소나 (PRD-02) | 2.0p | 6명 프로필 카드 (산업별 2명) |
| Section 3: 보이스봇 MVP (PRD-03) | 3.0p | 공통 코어 + 산업 애드온 + 예시 대화 |
| Section 4: Agent Assist MVP (PRD-04) | 3.0p | 통화 전/중/후 기능 + 보이스봇 연동 시나리오 |
| Total | 10.0p | -- |

**분량 초과 시 압축 우선순위:**
1. 예시 대화 시나리오 수 축소 (산업별 1개씩 -> 대표 1개로)
2. 페르소나 카드 서술 압축 (표 형태 유지, 셀 내용 간소화)
3. Executive Summary 축소 (0.5p -> 0.3p)

## Table Formatting Contract

### Persona Card Table (PRD-02)

Each of the 6 personas uses this exact table structure:

```markdown
### [산업] 페르소나 N: [역할명]

| 항목 | 내용 |
|------|------|
| **역할** | [직함, 보고 라인, 주요 책임] |
| **관심사/목표** | [핵심 KPI, 달성하려는 목표] |
| **페인포인트** | [현재 겪고 있는 핵심 고통 — Phase 1 시나리오 연결] |
| **구매 기준** | [솔루션 선택 시 가장 중요한 기준 3가지, 번호로 나열] |
| **예산 규모** | [연간 IT/CS 예산 범위, 의사결정 단위] |
| **현재 솔루션** | [사용 중인 솔루션명 + 구체적 불만 사항] |
| **성공 지표** | [도입 후 기대하는 정량적 변화] |
```

- 7개 항목 모두 필수 (하나도 빠뜨리지 않음)
- 페르소나 수: 금융(CXO + CS센터장), 보험(CXO + CS센터장), 이커머스(CXO + CS센터장) = 6명
- 각 페르소나당 약 1/3 페이지

### Feature Table (PRD-03 공통 코어)

```markdown
| 기능 | 설명 | 코어 밸류 연결 | 자산 활용 |
|------|------|---------------|----------|
| [기능명] (Day 1 필수) | [1-2문장 비즈니스 가치 중심 설명] | [핵심 가치와의 연결] | 기존 자산 / 기존 자산 확장 / 신규 개발 |
```

- 4개 컬럼 고정
- "코어 밸류 연결" 컬럼은 모든 기능이 "틀리면 안 되는 상담"과 어떻게 연결되는지 명시

### Feature Table (PRD-03 산업 특화)

```markdown
| 산업 | 특화 기능 | 규제/비즈니스 근거 | 자산 활용 |
|------|----------|-------------------|----------|
| [금융/보험/이커머스] | [기능명] | [관련 법규 또는 비즈니스 요구 1줄] | 기존 자산 / 기존 자산 확장 / 신규 개발 |
```

- 4개 컬럼 고정

### Feature Table (PRD-04 Agent Assist)

```markdown
| 기능 | 설명 | 상담원 경험 | 자산 활용 |
|------|------|-----------|----------|
| **[기능명]** | [비즈니스 가치 중심 설명] | [상담원이 실제로 보는/경험하는 것] | 기존 자산 / 기존 자산 확장 / 신규 개발 |
```

- 4개 컬럼 고정
- 워크플로우 단계(통화 전/중/후)별로 별도 표 또는 `####` 소제목으로 구분

### Example Dialogue Format (PRD-03/04)

```markdown
**예시 대화 ([산업]):**
> 고객: "[고객 발화]"
> 보이스봇: "[AI 응답 -- 구체적 데이터 포함]
> (출처: [문서명 제X조 제X항])"
> [상담원 화면: 출처 문서 하이라이트 표시, 해당 원문 링크]
```

- blockquote (`>`) 내에 대화 배치
- 화자 구분: `고객:` / `보이스봇:` / `상담원:` (bold 아님, 콜론으로 구분)
- 출처는 반드시 괄호 안에 구체적 문서/조항 표기
- 상담원 화면 표시는 대괄호 `[]` 안에 서술

## Data Citation Contract

Phase 1에서 확립한 인용 체계를 계승하고 확장한다.

### Phase 1 데이터 포인트 인용 (기존)

```markdown
> **[MKT-DP-XX]** {데이터 내용}: **{수치}** ({연도})
> -- 출처: {출처명} ({신뢰도})
```

- PRD 내에서 Phase 1의 MKT-DP-01 ~ MKT-DP-18을 직접 인용
- 최소 5개 이상의 MKT-DP 인용 필수 (PRD-01 비전 섹션에서)
- 인용 시 원본 수치를 그대로 사용, 재계산하지 않음

### Phase 2 신규 출처 (추가)

Phase 1의 각주 형식(`[^1]` ~ `[^9]`)을 계승하되, Phase 2에서 추가되는 출처는 본문 내 직접 인용 형태를 우선 사용한다.

```markdown
{본문 서술}(출처: {출처명}, {연도})
```

- 문서 말미에 별도 출처 목록 섹션은 두지 않음 (10페이지 제약)
- 각주가 필요할 경우 Phase 1과 번호 중복을 피해 `[^10]`부터 시작
