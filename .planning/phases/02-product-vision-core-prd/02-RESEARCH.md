# Phase 2: Product Vision & Core PRD - Research

**Researched:** 2026-04-02
**Domain:** AICC 제품 PRD 문서 작성 -- 제품 비전/포지셔닝, 바이어 페르소나, 보이스봇 MVP, Agent Assist MVP 기능 정의
**Confidence:** MEDIUM-HIGH

## Summary

Phase 2는 Phase 1의 시장 분석 데이터를 근거로, 뤼튼 AICC 제품의 정체성(비전/포지셔닝)과 두 핵심 제품(보이스봇, Agent Assist)의 MVP 기능을 확정하는 PRD 문서를 산출물로 한다. 코드 구현이 아닌 **기획 문서 작성** Phase이며, 독자는 PM 역량을 판단하는 리더십/경영진이다. 핵심 과제는 (1) Perplexity 비유를 활용한 구축형 vs SaaS 포지셔닝 서술, (2) 금융/보험/이커머스 6명의 프로필 카드형 페르소나 정의, (3) 보이스봇 공통 코어 + 산업 애드온 구조의 MVP 기능셋, (4) 상담원 워크플로우(통화 전/중/후) 기준 Agent Assist MVP 기능셋을 하나의 통합 문서에 10페이지 이내로 압축하는 것이다.

B2B SaaS PRD 작성에서 2025-2026년 표준은 "짧고 날카로운 실행 문서"다. 과거 50페이지짜리 사양서가 아니라, 문제-해결-기능-측정을 긴밀히 연결하는 간결한 문서가 경영진 대상 PRD의 표준이다. 특히 이 Phase의 산출물은 Phase 3(가격 전략, 우선순위)와 Phase 4(UX 와이어프레임)의 직접 입력이 되므로, 기능 정의가 모호하면 후속 Phase 전체가 흔들린다.

보이스봇과 Agent Assist의 MVP 기능 정의에서 핵심은 "할루시네이션 방지"와 "컴플라이언스 내장"이 Day 1 필수라는 점이다. 2025-2026년 AICC 시장에서 RAG 기반 출처 추적 응답은 차별화가 아닌 신뢰의 기본 조건으로 자리잡고 있으며, 금융위 AI 가이드라인 시행(2026 1분기)이 이를 규제적으로 뒷받침한다. "틀리면 안 되는 상담"이라는 코어 밸류가 기능 명세의 모든 항목을 관통해야 한다.

**Primary recommendation:** PRD-01~04를 하나의 통합 문서에 비전 -> 페르소나 -> 보이스봇 -> Agent Assist 순서로 구성하되, 각 기능 항목에 (1) 기능 설명, (2) 예시 대화/시나리오, (3) 기존 자산 활용 vs 신규 개발 표시를 포함하여 "이 제품이 무엇을 하는지" 즉시 이해 가능하게 작성할 것.

## Architecture Patterns

### 권장 문서 구조 (통합 PRD)

```
docs/prd-core.md (단일 통합 문서, 10페이지 이내)
├── Executive Summary (0.5페이지)
│   └── Perplexity 비유 hook + 핵심 메시지 3줄
├── Section 1: 제품 비전 및 포지셔닝 (PRD-01) (1.5페이지)
│   ├── 1.1 왜 지금인가: 시장 기회 (Phase 1 데이터 인용)
│   ├── 1.2 구축형 vs SaaS: 포지셔닝 전환
│   └── 1.3 Perplexity 비유: "출처 기반 신뢰성" AICC
├── Section 2: 타겟 고객 및 페르소나 (PRD-02) (2페이지)
│   ├── 2.1 금융 페르소나 (CXO + CS센터장)
│   ├── 2.2 보험 페르소나 (CXO + CS센터장)
│   └── 2.3 이커머스 페르소나 (CXO + CS센터장)
├── Section 3: 보이스봇 MVP (PRD-03) (3페이지)
│   ├── 3.1 공통 코어 기능 (STT/TTS, RAG, 할루시네이션 가드, 상담원 전환)
│   ├── 3.2 산업별 특화 모듈 (금융/보험/이커머스)
│   └── 3.3 예시 대화 시나리오
├── Section 4: Agent Assist MVP (PRD-04) (3페이지)
│   ├── 4.1 통화 전: 고객 정보 자동 로드
│   ├── 4.2 통화 중: 실시간 전사/요약/추천 응답/컴플라이언스 경고
│   ├── 4.3 통화 후: 요약 자동 저장/후처리 자동화
│   └── 4.4 보이스봇 연동 시나리오
└── 부록: 기능 요약 매트릭스 + Phase 1 데이터 포인트 참조
```

**분량 배분 근거:** 10페이지 제약 하에서 보이스봇(3p)과 Agent Assist(3p)에 가장 많은 분량을 할당하는 이유는, 이 두 섹션이 Phase 3(우선순위/자산 매핑)과 Phase 4(UX 와이어프레임)의 직접 입력이기 때문이다. 비전(1.5p)과 페르소나(2p)는 간결하되 임팩트 있게.

### Pattern 1: Perplexity 비유 프레이밍

**What:** 제품 비전을 Perplexity의 "출처 기반 신뢰성" 포지셔닝 전략과 유사한 프레이밍으로 서술
**When to use:** PRD-01 비전 섹션 초반, 비기술 바이어가 즉시 이해할 수 있는 한 줄 포지셔닝 필요 시
**핵심 원칙:**
- 비유의 범위를 "포지셔닝 전략의 유사성"으로 한정 (Phase 1에서 확립)
- 기능적/기술적 동등성 주장 금지
- "ChatGPT : Perplexity = 범용 LLM AICC : 뤼튼 컴플라이언스 AICC" 구도

**예시 서술:**
```markdown
범용 LLM 기반 AICC가 "자연스러운 대화"로 경쟁한다면,
뤼튼 AICC는 "이 답변이 어디서 왔는지"를 보여주는 것으로 차별화한다.
검색 시장에서 Perplexity가 출처 기반 신뢰성으로 새로운 축을 열었듯,
AICC 시장에서 "틀리면 안 되는 상담"에 특화된 새로운 카테고리를 정의한다.
```

### Pattern 2: 프로필 카드형 페르소나

**What:** 구조화된 카드 포맷으로 각 페르소나를 정의
**When to use:** PRD-02 페르소나 섹션, 6명 각각에 적용
**카드 구성 축:**

```markdown
### [산업] 페르소나 1: [역할명]

| 항목 | 내용 |
|------|------|
| **역할** | [직함, 보고 라인, 주요 책임] |
| **관심사/목표** | [핵심 KPI, 달성하려는 목표] |
| **페인포인트** | [현재 겪고 있는 핵심 고통] |
| **구매 기준** | [솔루션 선택 시 가장 중요한 기준 3가지] |
| **예산 규모** | [연간 IT/CS 예산 범위, 의사결정 단위] |
| **현재 솔루션** | [사용 중인 솔루션 + 불만 사항] |
| **성공 지표** | [이 제품을 도입하면 어떤 변화를 기대하는지] |
```

**핵심 원칙:**
- 의사결정자(CXO)와 실무자(CS센터장)의 관심사가 다름을 반영
- CXO: ROI, 리스크 감소, 전략적 포지셔닝
- CS센터장: 운영 효율, 상담원 만족도, 일상 업무 개선
- 경쟁사 언급은 "현재 솔루션" 항목에서 자연스럽게 포함

### Pattern 3: 공통 코어 + 산업 애드온 기능 구조

**What:** 보이스봇 기능을 모든 산업 공통 코어와 산업별 특화 모듈로 분리
**When to use:** PRD-03 보이스봇 MVP 기능 정의

```markdown
#### 공통 코어 (모든 산업 Day 1 필수)
| 기능 | 설명 | 자산 활용 |
|------|------|----------|
| STT/TTS 엔진 | 실시간 음성 인식 + 합성 | 기존 자산 |
| Hybrid RAG 응답 | 출처 추적 가능한 지식 기반 응답 | 기존 자산 |
| 할루시네이션 가드 | 다층 검증 (출처 매칭 + 신뢰도 스코어 + 규정 위반 탐지) | 신규 개발 |
| 상담원 전환 (에스컬레이션) | 처리 불가 시 인간 상담원 연결 + 맥락 전달 | 기존 자산 |

#### 산업별 특화 모듈
| 산업 | 특화 기능 | 설명 | 자산 활용 |
|------|----------|------|----------|
| 금융 | 상품 안내 컴플라이언스 | 투자상품 설명 시 약관 출처 자동 인용, 불완전판매 방지 | 신규 개발 |
| 보험 | 설명의무 체크리스트 | 보험 상품 안내 시 면책조항/보장범위 필수 설명 자동 추적 | 신규 개발 |
| 이커머스 | 정책 정확 안내 | 상품별 반품/교환/환불 정책 DB 연동, 정확한 정책 안내 | 기존 자산 확장 |
```

### Pattern 4: 상담원 워크플로우 기반 Agent Assist 기능 배치

**What:** Agent Assist 기능을 통화 전/중/후 워크플로우 단계에 배치
**When to use:** PRD-04 Agent Assist MVP 기능 정의

```markdown
#### 통화 전 (Pre-Call)
- 고객 정보 자동 로드: CRM/이전 상담 이력 자동 표시

#### 통화 중 (In-Call)
- 실시간 전사: 음성 -> 텍스트 실시간 변환
- 실시간 요약: 대화 핵심 내용 자동 요약
- 추천 응답: 맥락 기반 최적 응답 제안 (RAG 기반 출처 포함)
- 컴플라이언스 경고: 규정 위반 감지 시 실시간 알림 + 해당 규정 조항 표시

#### 통화 후 (Post-Call)
- 요약 자동 저장: 상담 요약 CRM 자동 기록
- 후처리 자동화: ACW(After Call Work) 시간 단축
```

### Pattern 5: 기능 명세 + 예시 대화 패턴

**What:** 각 주요 기능을 명세 후 실제 상담 예시 대화로 동작 시연
**When to use:** PRD-03/04의 주요 기능 설명 시

```markdown
#### 출처 추적 응답

**기능 설명:** 고객 문의에 대해 Hybrid RAG 기반으로 지식 DB에서 관련 정보를 검색하고,
응답과 함께 출처(약관 조항, 정책 문서, 규정 번호)를 명시한다.

**예시 대화 (보험):**
> 고객: "이 보험에 암 진단비가 포함되어 있나요?"
> 보이스봇: "네, 해당 상품에 암 진단 특약이 포함되어 있습니다.
> 1차 암 진단 시 2,000만원이 지급됩니다.
> (출처: 무배당 건강보험 약관 제15조 제2항)"
> [상담원 화면: 출처 문서 하이라이트 표시, 해당 약관 원문 링크]
```

### Anti-Patterns to Avoid

- **기능 나열만 하는 PRD:** 기능을 열거하되 "왜 이 기능이 필요한가"(페르소나의 페인포인트 연결)를 빠뜨리면 설득력 없음 -- 모든 기능을 페르소나의 문제와 연결
- **기술 중심 서술:** "Hybrid RAG 아키텍처는 Elasticsearch와 Neo4j Knowledge Graph를 결합하여..." -- 비즈니스 가치 중심으로 전환 ("상담원이 AI 응답의 근거를 즉시 확인할 수 있어 신뢰 기반 상담이 가능")
- **모호한 MVP 경계:** "추후 확장 가능" 같은 표현으로 MVP 범위를 흐리게 함 -- Day 1 필수 vs Phase 2 vs Future를 명확히 구분 (상세 우선순위는 Phase 3에서)
- **페르소나의 피상적 정의:** 역할명만 있고 구체적 페인포인트/구매기준이 없는 페르소나 -- 각 페르소나가 "현재 어떤 솔루션을 쓰고, 왜 불만인지"까지 서술
- **Phase 1 데이터 무시:** 시장 분석 없이 기능만 나열 -- MKT-DP-XX 참조로 시장 데이터와 기능의 연결 고리를 명시

## Don't Hand-Roll

| 문제 | 직접 만들지 말 것 | 대신 활용할 것 | 이유 |
|------|-------------------|---------------|------|
| 시장 데이터 | 새로운 시장 분석 | Phase 1 MKT-DP-01~18 인용 | 이미 검증된 18개 데이터 포인트 존재 |
| 경쟁사 약점 | 독자적 경쟁 분석 | Phase 1 Section 2.3 4대 약점 패턴 인용 | 동일 데이터 이중 분석 방지 |
| 산업별 페인포인트 | 새로운 시나리오 창작 | Phase 1 Section 3.3 시나리오 재구성 | 이미 논증된 금융/보험/이커머스 시나리오 활용 |
| 포지셔닝 논증 | 새로운 차별화 프레임 | Phase 1 포지셔닝 맵 + White Space 논증 인용 | 2축 맵에서 우측 상단 빈 공간 논증 완료 |
| 기존 자산 목록 | 자산 조사 | PROJECT.md Validated 섹션 인용 | 코어 엔진, Hybrid RAG, AdminPanel 등 이미 정리됨 |
| 바이어 페르소나 프레임 | 독자적 프레임워크 발명 | B2B SaaS 표준 페르소나 카드 (역할/페인/구매기준/예산) | 경영진에게 익숙한 표준 포맷 |

**Key insight:** Phase 2의 핵심 가치는 "새로운 분석 생성"이 아니라 "Phase 1의 시장 데이터를 제품 기능으로 번역하는 것"이다. 데이터를 새로 만들지 말고, Phase 1 데이터를 기능 정의의 근거로 활용하라.

## Common Pitfalls

### Pitfall 1: 비전과 기능의 분리
**What goes wrong:** 비전 섹션에서 "출처 기반 신뢰성"을 강조하지만, 기능 명세에서는 출처 추적과 무관한 일반적 AICC 기능만 나열
**Why it happens:** 비전은 전략팀이, 기능은 개발팀이 각각 작성하는 관행에서 비롯. 통합 문서에서도 이 단절이 발생 가능
**How to avoid:** 모든 기능 항목에 "코어 밸류 연결" 컬럼을 추가하여, 각 기능이 "틀리면 안 되는 상담"이라는 핵심 가치와 어떻게 연결되는지 명시
**Warning signs:** 기능 표에서 "출처", "컴플라이언스", "할루시네이션"이라는 단어가 한 번도 등장하지 않을 때

### Pitfall 2: 페르소나의 균일화
**What goes wrong:** 금융 CXO, 보험 CXO, 이커머스 CXO의 페인포인트가 거의 동일하게 서술 ("비용 절감", "효율성 향상")
**Why it happens:** 산업별 규제 환경과 비즈니스 모델의 차이를 반영하지 못하고, 범용적 B2B 고통점만 나열
**How to avoid:** Phase 1 Section 3.3의 산업별 시나리오(금융: 불완전판매, 보험: 설명의무 위반, 이커머스: 반품 오안내)를 페르소나의 핵심 페인포인트로 직접 연결. 각 산업의 규제/법적 리스크가 다름을 반영
**Warning signs:** 6개 페르소나의 "페인포인트" 항목이 교차 대체 가능할 정도로 유사할 때

### Pitfall 3: MVP 범위 팽창 (Feature Creep)
**What goes wrong:** "있으면 좋은" 기능이 Day 1 필수에 포함되어 MVP 범위가 비현실적으로 확장
**Why it happens:** PRD 작성 시 "경쟁사 대비 우위"를 보여주려는 욕구 + 10페이지 제약 하에서도 기능 목록을 최대한 많이 넣으려는 경향
**How to avoid:** Day 1 필수 기준을 "신뢰성 우선"으로 명확히 설정 (CONTEXT.md에서 확정). 출처 추적 + 할루시네이션 가드 + 상담원 전환이 Day 1 핵심. 나머지는 기능 표에 "Phase" 컬럼으로 구분하되, 상세 우선순위는 Phase 3(PRD-06)에서
**Warning signs:** MVP 기능 목록이 20개를 초과하거나, "Day 1 필수" 항목이 전체의 80% 이상을 차지할 때

### Pitfall 4: Phase 1 데이터 인용 없는 주장
**What goes wrong:** "시장이 빠르게 성장하고 있다", "경쟁사 대비 차별화된다"라는 주장을 Phase 1 데이터 인용 없이 서술
**Why it happens:** PRD를 독립 문서로 작성하려는 관성 -- Phase 1 산출물의 존재를 잊거나 참조하기 귀찮아 함
**How to avoid:** PRD 비전 섹션에서 최소 5개 이상의 MKT-DP-XX 데이터 포인트를 직접 인용. 문서가 독립적으로 읽혀야 하지만, 근거는 Phase 1에서 가져와야 함
**Warning signs:** PRD에 숫자가 거의 없거나, Phase 1의 18개 데이터 포인트 중 하나도 인용되지 않을 때

### Pitfall 5: Agent Assist와 보이스봇의 시너지 누락
**What goes wrong:** 보이스봇과 Agent Assist를 완전히 독립된 제품으로 서술하여, 두 제품 간 연동/시너지가 드러나지 않음
**Why it happens:** 각 제품의 기능 정의에 집중하다 보면 제품 간 연결점을 놓침
**How to avoid:** CONTEXT.md에서 확정된 "보이스봇 연동 시나리오"를 반드시 포함. 보이스봇 처리 불가 -> 상담원 전환 시 Agent Assist가 맥락 자동 인수하는 시나리오로 두 제품의 시너지를 명시적으로 시연
**Warning signs:** Section 3(보이스봇)과 Section 4(Agent Assist) 사이에 상호 참조가 없을 때

### Pitfall 6: 10페이지 제약 위반
**What goes wrong:** 기능을 상세히 명세하다 보면 문서가 15-20페이지로 팽창
**Why it happens:** Phase 1의 시장 분석서가 15-20페이지였으므로, PRD도 유사한 분량이 될 것이라는 무의식적 기대
**How to avoid:** 섹션별 분량 사전 할당 (Executive Summary 0.5p, 비전 1.5p, 페르소나 2p, 보이스봇 3p, Agent Assist 3p). 기능 상세는 표 형태로 압축, 서술은 핵심만
**Warning signs:** 초안 작성 중 한 섹션이 할당 분량의 1.5배를 초과할 때

### 패턴 1: PRD-01 비전 서술 (Perplexity 비유 + 데이터 인용)

```markdown
## 제품 비전: "틀리면 안 되는 상담"을 위한 컴플라이언스 AICC

한국 AICC 시장은 3조5,349억원 규모(MKT-DP-01 참조)로 사상 최대이며,
AI 컨택센터 소프트웨어 시장은 연평균 23.7~28.8% 성장 중이다(MKT-DP-02, 03 참조).
그러나 이 시장에서 "성과 기반 과금 x 컴플라이언스 특화"를 동시에 제공하는 플레이어는
국내외를 막론하고 부재하다.

**포지셔닝:**
범용 LLM 기반 AICC가 ChatGPT라면, 뤼튼 AICC는 Perplexity다.
ChatGPT가 "자연스러운 답"을 생성한다면, Perplexity는 "출처가 있는 답"을 찾아준다.
마찬가지로, 기존 AICC가 "더 자연스러운 대화"로 경쟁한다면,
뤼튼 AICC는 "이 답변이 약관 제12조에서 왔습니다"라고 보여주는 것으로 차별화한다.
```

```markdown
#### 4.2 통화 중 (In-Call) -- 4대 핵심 기능

|------|------|-----------|----------|

```

### 패턴 5: 보이스봇-Agent Assist 연동 시나리오

```markdown
#### 4.4 보이스봇 연동 시나리오

**시나리오: 보이스봇 -> 상담원 전환 시 맥락 자동 인수**

1. 고객이 보이스봇에게 복합 보험 상품 비교를 요청
2. 보이스봇이 2개 상품까지 안내 후, 3개 이상 비교는 복잡도 한계로 상담원 전환 판단
3. 전환 시 Agent Assist가 자동으로 수신:
   - 보이스봇 대화 전체 전사 텍스트
   - 고객이 언급한 상품 2개 + 비교 요청 키워드
   - 고객 감정 상태 (중립/긍정)
4. 상담원 화면에 "고객이 A상품과 B상품을 비교 중이며, 3번째 상품 추가 비교 요청" 요약 표시
5. 상담원은 처음부터 다시 묻지 않고 바로 상담 이어감

**시너지 효과:** 고객은 같은 말을 반복하지 않고, 상담원은 맥락 파악에 시간을 쓰지 않는다.
```

## Open Questions

1. **페르소나별 예산 규모의 현실성**
   - What we know: 한국 대형 금융지주의 IT 예산은 수백억~수천억원 규모. AICC 관련 예산은 전체의 일부
   - What's unclear: 중견 보험사나 이커머스 기업의 AICC 관련 구체적 예산 데이터가 공개되지 않음
   - Recommendation: 금융은 대기업 기준(50억원+), 보험은 중견 기준(10-30억원), 이커머스는 규모별 차이가 큼(5-50억원)으로 범위 추정. Phase 3(PRC)에서 가격 전략과 정합성 검증

2. **할루시네이션 가드의 구체적 구현 수준**
   - What we know: 다층 방어(RAG + NLI 검증 + 가드레일)가 최신 표준. Stanford 연구에서 RAG+RLHF+가드레일 결합 시 96% 할루시네이션 감소
   - What's unclear: PRD 수준에서 기술 아키텍처를 얼마나 상세히 서술할지 (독자가 경영진)
   - Recommendation: PRD에서는 "다층 검증으로 오답 리스크 원천 차단"이라는 비즈니스 가치로 서술하고, 기술 상세는 Phase 3(PRD-05 자산 매핑)에서. 다만 "출처 매칭 검증 -> 신뢰도 스코어 -> 규정 위반 탐지" 3단계 구조 정도는 언급

3. **컴플라이언스 경고의 규정 매핑 범위**
   - What we know: 금융위 AI 가이드라인 7대원칙, 보험업법 설명의무, 전자상거래법이 주요 규제
   - What's unclear: MVP 단계에서 매핑할 규정의 구체적 범위 -- 주요 법률의 핵심 조항만? 시행령/시행규칙까지?
   - Recommendation: MVP에서는 산업별 핵심 규정 3-5개만 매핑하고, 규정 DB는 고객사별 커스터마이징 가능하도록 설계. 상세는 Phase 3에서

4. **10페이지 제약과 6개 페르소나의 분량 균형**
   - What we know: 6개 페르소나를 각각 상세히 서술하면 페르소나만 3-4페이지 소요
   - What's unclear: 페르소나 깊이 vs 기능 명세 상세도의 최적 균형
   - Recommendation: 페르소나는 표 형태의 프로필 카드로 압축 (1 페르소나당 약 1/3 페이지), 기능 명세에 더 많은 분량 할당. 페르소나의 깊이보다는 "기능과의 연결"이 더 중요
